# Reaction to Jake Van Clief's Folder System (The Best AI Training Video Ever Made, Prove Me Wrong)

**The core thesis / claim**
The speaker argues that Jake Van Clief's file-and-folder "MD routing" system is the most valuable AI training resource on YouTube because it replaces fragile, app-specific agent memory with a durable, human-readable map of instructions. By nesting markdown files—agents.md, router.md, skill files—in a folder hierarchy, any LLM can become a context-aware agent that never gets lost, routes itself between projects, and maintains continuity across sessions and applications.

**How it builds**
The argument moves from personal testimony to live demonstration. First, the speaker runs a simple "control your agent" experiment—routing an LLM through multiple files to write the phrase "control your agent"—to prove the concept. He then de-mystifies what an "agent" actually is (just prompts and instructions), using OpenClaw's built-in agents.md as professional precedent. Next, he introduces the "map" metaphor: every folder is a floor in a building, and router.md files act as directories. He demonstrates real-world routing across his own projects (I Scale merch, Pin Twist), then shows the "pick-up / hand-off" protocol that lets him switch seamlessly between Claude, Codex, VS Code, Cursor, Hermes, and CLI tools without losing context. Finally, he reframes the system as a human organizational pattern, not an AI-specific trick, and urges viewers to start small and build their own.

**Key passages**
- "the information in this video has completely changed my experience with AI" [t0] — Personal stakes and the claim that this system eliminated months of frustration and lost work.
- "in your file and folder system you have full control over your agent" [t0] — The payoff of the opening experiment: deterministic control via written instructions rather than repeated prompting.
- "why not just have cloud code become the agent you need when you're working in the workspace I see that's a mic drop moment and some people don't even realize it what this man just said is incredible" [t0] — The central insight that the workspace itself, not a dedicated app, constitutes the agent.
- "this is a building of your life you have your personal stuff work stuff education family it all goes here and every part of your life is a different floor your career your print on demand business every folder is another department" [t0] — The map metaphor that makes the routing system intuitive and scalable.
- "it doesn't matter where I go the agent is the same it doesn't matter if it's Claude Codex or these local LLMs" [t0] — Evidence that the instruction file, not the vendor or app, defines the agent's behavior.
- "this is not an AI thing this is a human that can read this... you're really not learning AI things you're learning how to route anything that reads it" [t0] — Reframing the technique as universal human organization rather than specialized AI tooling.

**Themes**
- **Map & Router**: Agents must have a written map (router.md) so they always know where they are, where to go next, and when to route back.
- **Agent as Prompts**: An "agent" is not a special product; it is simply a set of instructions read by an LLM, as evidenced by OpenClaw's own agents.md.
- **App/LLM Agnosticism**: The same agents.md file makes any tool—Claude, Codex, Cursor, Hermes, CLI—behave identically, freeing the user from vendor lock-in and model drift.
- **Pick-up / Hand-off**: Codified session continuity protocols let work persist in files rather than in a model's ephemeral context window, enabling seamless cross-app workflows.
- **Human Organization**: The system is fundamentally a human routing protocol that happens to be executed perfectly by AI; it is organizational design, not prompt engineering magic.

## Tensions & objections
- **Maintenance Overhead vs. Native Memory**: The system shifts the burden of context management from the model to the user. Maintaining `router.md`, `agents.md`, and manual "hand-off" files requires constant upkeep. As LLMs improve native long-term memory and project context (e.g., Claude Projects, ChatGPT Memory), this manual file-routing may become obsolete overhead rather than a superior alternative.
- **Fragility of Multi-Step File Chaining**: The "control your agent" experiment and the "pick-up" protocol rely on the LLM flawlessly executing a chain of file reads and writes. In practice, LLMs frequently skip steps, hallucinate file contents, or hit context limits when navigating deep folder structures, making the "never gets lost" claim aspirational rather than guaranteed.
- **The "20% Understanding" Caveat**: The speaker explicitly admits he is only at "about 20% of understanding Jake's system" and is showing a "dumb down version." This implies the actual system is vastly more complex, raising the barrier to entry and suggesting it may be over-engineered for typical use cases.
- **Contradiction on AI's Organizational Ability**: The speaker claims "AI is not good at organizing... you need to be in charge," yet earlier demonstrates using an AI ("workspace system" agent) to architect and build the entire folder structure and routing rules. The system is ultimately AI-generated organization, just externalized into markdown files.
- **Conspiracy vs. Pragmatism on Model Degradation**: The speaker claims companies intentionally make models dumber and that this system exposes it, allowing users to save money by jumping between cheaper tiers. However, this conflates normal model variance or context-window limitations with malicious degradation, and the "cost savings" rely on the user spending significant unpaid labor to maintain the routing infrastructure.

## What's worth learning — and what we could do with it
1. **Create a single `agents.md` in your active project root** — Write a one-page identity card for the LLM you want (e.g., "You are a careful code reviewer who checks for security issues and tests"). Reference it explicitly at the start of every new chat/session in that project. This costs zero infrastructure and immediately makes context portable between Claude, ChatGPT, Cursor, etc.
2. **Build a two-level `router.md` prototype** — Pick one domain (e.g., "content creation" or "software project") and create a folder with a `router.md` that lists sub-folders and what each contains. Ask an LLM to read the router, then ask it to route itself to a specific sub-task. Measure whether it reaches the right file on the first try. If it fails, the router instructions need sharpening, not the model.
3. **Codify one "hand-off" protocol** — Before ending a session, paste a structured summary into a `session-log.md`: current goal, completed steps, next step with file path, and open questions. In the next session (even in a different app), feed that file first. Test whether you recover context faster than scrolling through old chat history.
4. **Audit your current AI workflows for "lost context" waste** — For one week, note every time you repeat instructions, re-explain a project, or abandon a chat because the model "forgot." If the count is >3, the folder-map overhead is likely justified; if <1, native memory may be sufficient.
5. **Treat the folder system as a human-readable contract, not an AI trick** — Write `router.md` and skill files so a new human teammate could onboard from them. If a human can't follow the map, an LLM won't reliably either. This keeps the system maintainable even as models improve.
6. **A/B test against native project memory** — Run the same complex multi-step task twice: once with your file-routing system, once with Claude Projects/ChatGPT Memory/Cursor composer. Time both, count corrections needed, and note where the native system hallucinates or drifts. Use the result to decide which contexts you route manually and which you delegate to vendor memory.
