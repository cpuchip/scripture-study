# Phase 3a — Watchman pass with model dispatch

Phase 3 was the next real frontier after 2.7a. The user said *"let's get the model dispatch sidecar working so we can unblock some of this earlier work"* and named pg-ai-stewards as the project. The literal Phase 3 in `phases.md` is huge (pipelines, sidecars, MCP, web UI, scripture-study POC). Same trap as 2.7. Same fix: split honestly.

**Phase 3a = minimum unblocker.** Just enough model dispatch to make `stewards-cli watchman pass` work. Single provider, no MCP, no UI, no pipelines. Ship the discipline; defer the edifice.

## What I almost got wrong

My first instinct was to scope around LM Studio because it's local and free. The user corrected: *"OpenCode Go/Zen to do the heavy lifting in the DB with Kimi K2.6. it's powerful and reasonably cheap."* Right call — the tokenomics doc had already named kimi as the bulk-work model. LM Studio is the local backstop, not the default.

The user also nudged: *"you might want to look over our journals/history to see everything we've done!"* And they were right. The 2026-05-03 Phase 1.6 journal documented a working agent loop with kimi-k2.6 via opencode_go — `brain_search_text` + `skill` tool calls, tool replies, continuation chat, finish_reason=stop, 18 seconds, $0.0005. The chat path was **proven** months before I started this session.

So why did the live DB show zero messages? The DB was rebuilt yesterday for the 2.5/2.6 work. Code intact, schema intact, agents+skills+tool_defs seeded — but no chat had ever run on this volume. Code-state vs. data-state were diverged, and I almost wrote net-new dispatch code I didn't need.

Reading before assuming. The user's nudge to check journals saved a half-day of redundant work.

## What shipped

**SQL migration:** `extension/3a-watchman-pass.sql`. Adds:
- `watchman-consolidator` agent family (default + kimi-* variants, same prompt, temp=0, steps=1, NO tools)
- `agent_tool_perms ('watchman-consolidator', '*', 'deny')` — structural enforcement so `compose_tools` returns `[]` for this agent
- `stewards.watchman_input(slug)` — composes the user message: doc body + 1-hop graph neighborhood from `context_for(slug, 1)`

**System prompt** for watchman-consolidator:
- Five verdicts: `clean | drift | done | superseded | skipped`
- Hard rule: "you see ONLY what is provided. Do not pretend to know facts about files, code, or context outside the input."
- Strict JSON output, first character must be `{`
- Strong steer toward `skipped` rather than guessing
- Strong steer toward `drift` for any internal contradiction across doc + neighbors

**Go orchestration:** `WatchmanPass` in `cmd/stewards-cli/internal/show/show.go`. CLI: `stewards-cli watchman pass [--provider opencode_go] [--model kimi-k2.6] [--limit N] [--timeout 180] [--slug X] [--dry-run]`. Loop:
1. Build pass_id from timestamp
2. Pull dirty_queue ordered by updated_at ASC, LIMIT N (or use --slug for single doc)
3. For each doc: insert session → call `watchman_input(slug)` → `chat_enqueue` → poll work_queue.id at 250ms → read assistant message → parse JSON (defensive markdown-fence stripping) → `record_verdict` (+ `record_finding` if non-clean)
4. Print table summary at end

The bgworker stays generic. All watchman-specific semantics live in the CLI for 3a. When 2.7b lands, this same logic moves into a Postgres bgworker scheduled pass — the CLI version proves the loop and lets 3a ship without a Rust rebuild.

## The first real verdict

`phase-pg-ai-stewards-0` (small phase doc, 280 chars). Dry-run produced:

```json
{
  "verdict": "skipped",
  "reasoning": "The document asserts Phase 0 is 'Done' and references external artifacts (scratch.md, probe/RESULTS.md, proposal.md), but none of these files appear in the provided 1-hop neighborhood. Without visibility into those deliverables, I cannot verify whether the stated completion status matches reality.",
  "finding": {
    "kind": "drift",
    "severity": "medium",
    "message": "The document's 'Done' claim depends on unverified external artifacts that were not included in the review neighborhood.",
    "suggested_action": "Include the referenced files (scratch.md, probe/RESULTS.md, proposal.md) in the next consolidation review so the Watchman can verify the deliverables."
  }
}
```

This is exactly the discipline the prompt asked for. Kimi didn't pretend. It admitted what it couldn't see AND told us what would help (1-hop graph isn't enough; we need to read the linked files for real spec-vs-claim comparison). That's a 3b agenda item *generated by the substrate itself.*

Re-run live (non-dry) at temp=0 produced `verdict: done` instead — kimi observed "Phase 0 — Done" in the title and went terminal. Both verdicts defensible; the variance at temp=0 with reasoning models is a known wart, not a bug. For 3b we may want to enforce structured output via `response_format: {type:"json_object"}`, which means injecting it into the body that `chat_post_internal` builds — schema change to `agents` table or to `dry_run_chat`. Deferred.

2. Ran `stewards-cli watchman pass --slug phase-pg-ai-stewards-0` → kimi-k2.6 returned `verdict: done`, 1m20s, 734 tokens in / 3861 out
3. **Doc disappeared from dirty_queue** — confirmed via `dirty_queue` view returning 0 rows for that slug

5. `stewards.verdicts` count by actor: `human=2, watchman=2`. Real model output, real token counts, real audit trail.

Suppression discipline (already verified yesterday in 2.7a) confirmed today by negation: the proposal returned to dirty_queue after yesterday's `ack` test, exactly as designed. No new mechanism needed; the substrate's anti-loop NOT EXISTS clause continues to enforce single-pass-per-finding.

## What didn't work (yet)

**Big docs hit the bgworker reqwest 120s ceiling.** Two scratch files (.scratch-abinadi-how-to-read, .scratch-art-of-delegation, both ~50KB+) and the proposal itself timed out:

```
POST https://opencode.ai/zen/go/v1/chat/completions: error sending request for url
```

This is a real Phase 3a limitation. The bgworker's `reqwest::blocking::Client::builder().timeout(Duration::from_secs(120))` was set for embedding latency. Kimi reasoning over a 50K input takes longer. Two paths forward (3b candidates):
1. Trim the input — feed only first N + last M characters (loses middle context but works for everything)
2. Bump the bgworker timeout via env var override (clean, but needs a new extension binary)

For 3a, we accept the limitation and document it. The discipline is proven on docs that fit the timeout window, which is most of them.

**First chat smoke after agent_tool_perms wasn't set.** Kimi reflexively called `brain_search_text` on the first turn (we'd left tools allowed). Steps=1 stopped the loop, but content was empty (the tool_calls assistant turn is empty content by OpenAI convention) and the JSON parse failed. Adding `('watchman-consolidator', '*', 'deny')` to `agent_tool_perms` fixed it. Important lesson: `compose_tools` is allow-by-default; new agents need explicit deny rules unless they need tools.

## Carry forward

- Bgworker 120s reqwest timeout is the real ceiling. Bump or trim for big docs.
- `response_format: {type: "json_object"}` injection would harden JSON output; needs schema change to agents or dry_run_chat.
- Variance at temp=0 with reasoning models is real. Live with it for now.
- Watchman verdicts are recorded in stewards.verdicts with full audit (actor, model, tokens, pass_id). Tokenomics is queryable today.
- The model itself surfaced a 3b agenda item: include linked files in the review neighborhood, not just graph-1-hop slugs.
