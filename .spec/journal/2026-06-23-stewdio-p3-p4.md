## P3 — persist a book's source text so a book chat quotes the BOOK

The gap (found in research): YouTube persists its source (`yt_transcripts`), books
discarded theirs (fetched at build, lived only in that run's messages). So a book
chat could ground on the doc + sessions + citations but not the book's own passages.

**Placement call:** the YT corpus is an *example* (`examples/yt-transcripts.sql`),
not core — and the book digester is too. So the book corpus went to a sibling
**`examples/book-corpus.sql`**, with the `book_search` grant to `work-item-chat`
living alongside the tool. No core chain bump (stays 00→45). Mirrors YT exactly.

**The one real design question:** how does the digester persist a *large* fetched
book without the model re-emitting it as a tool arg? YT reads a file server-side by
video_id; books come from `fetch_url`. Answer: a big `fetch_url` result lives in
`stewards.messages.content`, resolvable by `context_handle` — the same **page-in
handle** the build stage already uses with `result_read`. So `book_persist_corpus`
takes the handle (4–8 hex), reads the full content server-side, chunks, upserts.
The model passes a handle, never the text.

**The other design question:** semantic vs FTS. Confirmed the substrate has **no
synchronous query-embed** at tool-call time (`doc_search` is FTS; `doc_similar` uses
*precomputed* cosine edges). So the live `book_search` path is FTS (verbatim +
ranked passages with chunk_idx as location) — exactly like `transcript_search`. The
`book_chunks` embedding column + HNSW index are present but **opt-in** (config
`book_corpus_embed_chunks`, default OFF) — forward-looking infra for a future
precomputed-similarity feature, not a live path. Honest, not speculative-on.

Built: `book_text` (full_text + tsv), `book_chunks` (tsv + dormant embedding),
`book_persist_corpus(+tool)`, `book_search` (tolerates the doc slug `book-on-liberty`
OR `on-liberty`). Digester BUILD stage calls persist after fetch_url; `book_publish`
frontmatter gains `source_url` + `has_corpus` (to_regclass-guarded so the digester
stays standalone).

**Proof (live + local rig, $0):** persisted On Liberty (220 chunks). The Stewdio
chat, asked "where does Mill state the harm principle?", **called `book_search`,
found chunk 32, and quoted the book verbatim with the location** — British spelling
"civilised" and all. Inverse hypothesis held twice: the wrong-spelling American
"civilized" correctly returned `found_verbatim=false` (a quotation-mark guard
working), and "tyranny of the majority" matched verbatim at chunk 23.

## P4 — polish

- **Layout serialization** (Stewdio.vue): dockview `toJSON`/`fromJSON` ↔ localStorage,
  debounced save on change, restore on ready, "⟲ layout" reset. Reload-proven: 3
  groups restore, no console errors.
- **Multi-session history**: `GET /api/chat/sessions` (sessions for a target, by
  prefix, with preview/last-active/count) + a 💬 sidebar + ＋New + per-target
  last-session remembered in localStorage.
- **Provenance chips**: the SSE stream now carries each turn's tool NAMES; the panel
  maps them to facet chips (doc / book / corpus / sessions / citations / work item)
  on the retrieving rows — you see which facet grounded an answer. Plus an optimistic
  user echo so the compose latency doesn't read dead.

## The bug P3 surfaced (and the honest diagnosis)

The book chat 500'd through the API while a direct `dispatch_chat_turn` was fast.
Diagnosis: NOT history length (the session had 2 messages) — `chat_enqueue` →
`dry_run_chat` → `compose_system_prompt` composes the prompt + tool schemas **inline**
for a tools-on agent, ~7–11s, intermittently over the `/api/chat/send` 10s ceiling.
This was the same "transient 500" logged in P1. Fix: 10s → 45s (the reply still
streams over SSE; the send is just the enqueue). The deeper fix — defer composition
to the bgworker — is bigger than P4 and noted as a follow-up.

Also caught (verify-under-real-conditions earned its keep again): the chat agent
passed the *doc* slug `book-on-liberty` to book_search before reading frontmatter,
missing the `on-liberty`-keyed corpus and giving up. Fixed in the tool (strip a
leading `book-` when the literal slug misses) rather than relying on model behavior.

## Carry-forward (none blocking)

- Book backfill only covers books whose text gets re-fetched — existing book docs
  show `has_corpus=false` until re-digested (only On Liberty persisted so far).
- Chat-enqueue compose latency is papered over by the 45s timeout; the real fix is
  to defer body-composition to the bgworker.
- `book_chunks` embeddings stay dormant until a `book_similar` precomputed-edge
  builder exists (the `doc_similar` pattern).
- The work-item-chat local model over-explores (book_search ~12× before answering —
  correct answer, just chatty).
- A fully NL-chat-driven pipeline launcher is still a future enhancement beyond P2.
