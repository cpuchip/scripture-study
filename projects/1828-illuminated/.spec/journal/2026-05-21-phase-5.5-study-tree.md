### Study tree
- `useStudyTree.ts`: module-scoped Map<id, StudyNode> + activeNodeId.
  Five node kinds: `word`, `verse`, `chapter`, `render`, `note`.
  `visit(payload)` is the one mutating entry point — idempotent (active
  is identity = no-op; child with matching identity = reuse; otherwise
  new child). `navigateTo(id)` walks without creating. `removeSubtree`
  drops a node and all descendants. `clearAll` wipes.
- Persistence: localStorage `study-tree-v1` with 250ms debounce. Restore
  on first composable access.
- Markdown export: `exportTreeMarkdown(rootId?)` returns indented
  bullets + verse blockquotes + render excerpts. Wired to the panel's
  "Export ↻" button (clipboard-write).

### Cross-domain wiring
- `WordDetail.vue`: `watch(word, w => studyVisit({kind:'word', word:w}),
  {immediate:true})`. Every word-detail visit captures.
- `useWordData.selectWord(w)`: dynamic-imports useStudyTree and calls
  `visit({kind:'word', word})`. This is the cross-domain payoff —
  clicking a tier-word inside any rendered passage (canon chapter,
  demo verse, paste, even modernized render output) branches the tree.
- `VerseExplorer.fetchCanonChapter`: visits a chapter node after
  successful load. Range becomes a chapter node with `range` set.
- `useLLMRender.render`: on successful return, visits a render node
  with `sourceText` + `modernized` + `model` + `provider`. The
  modernized text is itself a passage — clicks within it (via
  HighlightedText → selectWord) would branch from this render node.

## Why this matters

Michael's framing — "five-dimensional time-travel chess" — names what
becomes possible: the reader walks `school → 1828 entry → "this verse
uses it" → Helaman 12:3 → click "endure" → endure's entry → branch back
two nodes → click "magnify" instead → ...`, and every step persists as
a navigable graph.

Cross-references in scripture study aren't linear. The tool now matches
the shape of how depth-of-study actually works. Not a breadcrumb — a
map you can re-traverse, branch off, and toggle between paths.

## What got surfaced but not built

- **Render-result re-render protection.** The render node payload holds
  the modernized text. If a user re-renders the same verse with the
  same model, identity dedupes — no new node, no second LLM call from
  the tree's perspective. But the actual render call happens in
  `useLLMRender.render()` regardless. Future: have the panel check
  if a render node exists and surface "you've rendered this; load
  cached?" rather than retriggering the LLM.
- **Stale-node ordering on browser back.** Walking a path via browser
  back doesn't update activeNodeId (only surface clicks do). Walking
  forward from a back-history position creates a new branch off the
  current active, NOT off the surface-displayed node. This is
  philosophically correct (tree = study graph, browser = view history)
  but might confuse some readers. Watch for feedback.
- **Tree → substrate study-write.** A completed tree is a natural
  `study-write-*` pipeline input. Future: "Build study from this tree"
  → POST tree JSON → substrate generates a synthesis study from the
  traversal. Bridge between exploration and articulation.
